### Designs are flexible, not contractual

**Important guidance for Claude Code:** the Figma designs are a strong, brand-aligned starting point — they should drive layout, hierarchy, and visual treatment. But **the founder has explicitly said the designs are flexible to be upgraded and improved during build.** This means:

- **Hard constraints (do not deviate):** the design tokens (`palette` and per-mode `themes`), brand voice from `product-vision.md` § Voice & Tone Guide, the brand anti-patterns, and the accessibility floor (WCAG AA).
- **Soft constraints (start from Figma, propose improvements where warranted):** specific layout choices, microcopy variations, the number of steps in a multi-step flow, the exact composition of form fields, and small interaction details.
- When you propose a deviation, note it in the PR description so the founder can review the design judgment.
- When the PRD and Figma disagree (e.g., auth methods, screen composition), default to the Figma for visual decisions and to the PRD for technical decisions (data model, API shapes, integration patterns) unless the trade-off is significant — in which case flag it as an Open Question.

## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can demo what you've built.
2. **Infrastructure before features.** Phase 0 sets up everything so feature work in later phases is fast and clean.
3. **Magic moment first.** The core value (live reactions during reading) lights up by the end of Phase 4 — we don't wait until the polish phase to feel the product.
4. **Test as you go.** Don't save testing for the end. Every task includes a verification step.
5. **Progressive enhancement.** Start with the simplest working version of each feature, then layer polish in Phase 7.
6. **Review before proceeding.** Push each completed phase as a PR and let CodeRabbit (or another review agent) review it before starting the next phase. External review catches issues the coding agent will miss.
7. **Ruthless scope discipline.** If a task isn't on this list, it's not in MVP. Open a Phase 2+ doc instead — don't sneak features in.
8. **Mobile-only at this stage.** Web companion is Phase 2 (post-MVP). Don't build for breakpoints we won't hit.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the beginning of each phase.
2. **Read selectively:** Each phase lists Reference sections — only those parts of the PRD and vision doc are needed for that phase. The agent should not read entire documents.
3. **Let it work:** The agent reads the roadmap, finds the first unchecked task, implements it, and marks it complete by changing `- [ ]` to `- [x]`.
4. **One session = one phase (ideally):** Try to complete a full phase in one session. If you stop, the agent can resume from the last unchecked task.
5. **Push a PR for review:** When a phase is complete, push the work as a PR and let CodeRabbit (https://coderabbit.ai) review it before starting the next phase.
6. **Need more context?** If a task references a section not in the phase's Reference list, the agent reads just that section on demand.

### Session Tips

- **Don't read everything:** The PRD and vision doc are large. Each phase's Reference sections tell the agent exactly what to read. Loading the full documents wastes context.
- **Don't skip tasks:** Tasks are ordered intentionally. Skipping creates dependency issues.
- **Verify after each phase:** Run the app on a real device after completing a phase to confirm everything works before moving on. The Verify steps in each task are the floor, not the ceiling.
- **Review before moving on:** Push a PR for each completed phase and let CodeRabbit check it. Don't start the next phase until the PR is merged.
- **Update the status line:** After completing tasks, update the header status: `**Status:** X/96 tasks complete` and `**Current Phase:** Phase N`.

### Prompt Templates

**Starting a new phase:**
> "Read docs/product-roadmap.md and find the current phase. Read only the Reference sections listed for that phase from docs/prd.md and docs/product-vision.md. Start working on the first unchecked task. After completing each task, update the checkbox to [x] in the roadmap file. Continue through the phase."

**Resuming after a break:**
> "Read docs/product-roadmap.md. Find where we left off (first unchecked task). Read only the Reference sections listed for the current phase from docs/prd.md and docs/product-vision.md. Continue from the first unchecked task."

**After completing a phase:**
> "Phase [N] is complete. Create a branch called phase-{N}/{slug}, commit all work, push, and open a PR targeting main. Title it 'Phase {N}: {Title}' and include the phase goal and completed task count in the body."

**Fixing an issue:**
> "There's a problem with [description]. Read the relevant section of docs/prd.md for the expected behavior and fix it. Don't mark any new tasks complete until the fix is verified."

### Phase Review Workflow

Each phase ends with a PR that gets reviewed by CodeRabbit (or another review agent of your choice — GitHub Copilot review, Codacy, etc.) before merging.

**Branch naming:**
- `phase-0/foundation-and-setup`
- `phase-1/auth-and-profile`
- `phase-2/clubs-and-memberships`
- `phase-3/book-upload-and-reader`
- `phase-4/live-reactions`
- `phase-5/creator-publishing-and-notifications`
- `phase-6/pro-tier-and-discovery`
- `phase-7/polish-and-launch`

**PR format:**

```
Title: Phase {N}: {Phase Title}

Body:
## Goal
{The phase's goal statement from the roadmap}

## Completed
- {X} tasks completed (TASK-{first} through TASK-{last})

## What to verify
- {Key things to test manually after merging}

## Reference sections used
- PRD: § ...
- Vision: § ...
```

**If CodeRabbit flags issues:**
1. Address feedback in follow-up commits on the same branch.
2. Let CodeRabbit re-review.
3. Merge only when the review is clean.

If you're not using GitHub or want to skip PR review: that's fine — the per-task verification still applies. You'll lose the catch-net for issues the coding agent missed, but you can always layer review back in later.
